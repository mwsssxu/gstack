# 第五章：偏执专家——如何像资深工程师一样审查

## 为什么通过测试的工作仍会失败

2025 年，有个团队来找我："我们的工作 100% 通过测试，但在实际环境崩溃了。"

我看了他们的工作。测试确实都通过了：
- 单元测试覆盖所有函数
- 集成测试覆盖所有 API
- E2E 测试覆盖所有用户流程

但**偏执专家**发现了问题：

- **N+1 查询**：循环中调用数据库，测试数据只有 3 条记录，实际有 10,000 条
- **重复操作问题**：多个请求同时修改同一数据，测试时只考虑了单个请求
- **信任边界**：直接使用 AI 输出写入数据库，没有验证
- **选项遗漏**：新增了新的状态选项，但程序没有处理所有可能的情况

**这不是"测试不足"，是"测试错了地方"。**

在智能工作系统里，**偏执专家**的角色就是找到这些**结构性缺陷**。

### 结构性缺陷 vs 风格问题

传统工作审查关注什么？
- 命名是否清晰
- 函数是否太长
- 注释是否充分
- 工作是否 DRY

这些都是**风格问题**。重要，但不是致命的。

**偏执专家**关注**结构性缺陷**：
- 会让系统在实际环境失败的问题
- 通过测试但实际不可靠的问题
- 在特定条件下才会暴露的问题

**结构性缺陷是沉默的杀手。**

### 八大结构性缺陷类别

**偏执专家**系统化地检查八大类别：

#### 1. 数据安全
- **N+1 查询**：循环中的数据库查询
- **过时读取**：缓存未失效导致的数据不一致
- **缺失索引**：大数据量下的性能杀手

#### 2. 重复操作与并发问题
- **双重提交**：重复点击导致重复操作
- **覆盖写入**：并发修改导致数据丢失
- **死锁**：资源竞争导致系统挂起

#### 3. AI 输出信任边界
- **JSON 格式错误**：AI 返回畸形 JSON
- **幻觉值**：AI 编造不存在的数据
- **提示注入**：用户输入被当作指令执行

#### 4. 选项与值的完整性
- **新增选项**：程序没有处理新添加的选项
- **状态转换**：非法状态转换未阻止

#### 5. 条件副作用
- **if/else 分支**：某些分支有副作用，某些没有
- **早期返回**：某些路径释放资源，某些不释放

#### 6. 硬编码值与字符串依赖
- **写死的数值**：直接写在代码里的数字、网址、接口地址
- **字符串比较**：字符串常量分散各处

#### 7. 无用代码与一致性
- **未使用代码**：函数、变量、导入
- **过时注释**：注释与代码不一致

#### 8. AI 提示问题
- **提示泄露**：敏感信息在提示中
- **信息溢出**：提示太长被截断

### Fix-First 审查：不只是发现问题

**偏执专家**最强大的设计是**Fix-First**（修复优先）。

不是只列出问题，而是**立即修复能修复的**。

#### 自动修复

对于机械性问题，**偏执专家**直接修复。

#### 询问用户

对于需要判断的问题，**偏执专家**询问用户。

**规则：** 涉及业务逻辑、安全、架构的问题询问用户。

### 实战案例：支付系统的审查

经过**偏执专家**的审查，发现关键问题并自动修复，最终结果：
- 重复收费归零
- 支付失败有明确消息
- 异常分类处理
- 状态机完整

### 我的建议：如何最大化偏执专家的价值

#### 建议 1：不要跳过 Fix-First

即使你觉得"这个简单，我自己修"。

**让**偏执专家**自动修复。** 这样你能专注于真正需要判断的问题。

#### 建议 2：认真对待范围检查

确保你构建了正确的东西，不仅仅是构建得好。

#### 建议 3：考虑第二意见

不同的 AI 有不同的盲点。

**两个 AI 审查比一个更可靠。**

### 结语：偏执的价值

大多数人认为"偏执"是负面词。

在工作中，**偏执是美德**。

**偏执专家**编码了这种偏执：
- 假设每个 API 调用都会失败
- 假设每个用户都会恶意使用
- 假设每个并发场景都会发生
- 假设每个边界情况都会被触发

**这不是悲观，是专业。**

在 AI 时代，工作的边际成本趋近于零。

**偏执的价值被放大了 1000 倍。**

因为一次实际事故的成本远超 1000 倍的预防成本。

所以，拥抱偏执。

运行**偏执专家**。
