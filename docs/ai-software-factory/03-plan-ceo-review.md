# 第三章：/plan-ceo-review——像 CEO 一样思考战略

## 从"功能实现"到"产品战略"的跃迁

有个创始人告诉我："我要加个图片上传功能。"

听起来很合理，对吧？用户需要上传图片，那就加个文件选择器，保存到服务器。

但`/plan-ceo-review`问了个不同的问题：

**"用户真正要的是什么？"**

也许不是"上传图片"，而是"快速创建一个能卖出去的商品列表"。

如果是这样，整个方案就变了：
- 能从图片识别产品吗？
- 能自动搜索网络信息完善描述吗？
- 能建议最佳封面图吗？
- 能检测图片质量并给出改进建议吗？

**这就是 CEO 思维：不是实现功能，是交付价值。**

### 为什么需要 CEO 审查

在软件公司，最常见的错误是什么？

**工程师思维主导产品决策。**

工程师思维是：
- "用户说要 X，那我就做 X"
- "这个功能需要 Y 技术，所以用 Y"
- "竞争对手有 Z，所以我们也要有 Z"

CEO 思维是：
- "用户真正的问题是什么？"
- "什么方案能交付 10 倍价值？"
- "我们的独特优势在哪里？"

在传统公司，这两种思维由不同的人负责：
- 产品经理/CEO 负责"做什么"和"为什么"
- 工程师负责"怎么做"

但在 AI 时代，一个人可以同时拥有两种思维。

`/plan-ceo-review`就是把 CEO 思维编码成 AI。

### 四种模式：因势而动的战略选择

`/plan-ceo-review`最强大的设计是**模式选择**。

不是所有计划都需要"做大做强"。有时候应该"保持范围"，有时候应该"做小做精"。

系统会根据上下文自动建议默认模式，但你永远有选择权。

#### 模式 1：范围扩张（SCOPE EXPANSION）

**何时使用：** 绿地项目、新功能、产品早期

**思维模式：** "如果我们有无限时间和资源，会做什么？"

**关键问题：**
- 10 倍检查："有什么版本能 deliver 10 倍价值，但只花 2 倍努力？"
- 理想状态："如果世界上最好的工程师有无限时间，会做成什么样？"
- 惊喜机会："哪些相邻的 30 分钟改进会让用户觉得'哇，他们想到了这个'？"

**输出：** 愿景文档 + 范围提案（每个都作为独立决策询问用户）

**实战案例：**

```
计划：添加用户认证功能

/plan-ceo-review (扩张模式):
"让我们想想 10 倍版本。

不是'加个登录表单'，而是：
- 无密码登录（魔法链接、生物识别）
- 社交登录一键集成（Google、GitHub、微信）
- 企业 SSO 自动配置
- 欺诈检测（异常登录地点、设备指纹）
- 自动会话管理（记住设备、智能过期）

每个都是扩张选项。你想要哪个？"
```

**我的经验：** 扩张模式最容易让人兴奋，但也最容易过度设计。所以要每个选项都询问用户。

#### 模式 2：精选扩张（SELECTIVE EXPANSION）

**何时使用：** 功能迭代、现有系统改进

**思维模式：** "保持当前范围，但看看有什么诱人的可能性"

**关键问题：**
- 复杂度检查："如果超过 8 个文件或 2 个新类，是味道吗？"
- 最小集："什么是最小变更集能实现目标？"
- 扩张扫描（但不加入范围）：
  - 10 倍版本是什么？
  - 哪些 30 分钟改进能增加惊喜？
  - 有平台潜力吗？

**输出：** 基线计划 + 精选选项（用户逐个决定）

**与扩张模式的区别：**
- 扩张模式："我们应该做这些！"
- 精选模式："这些是可能的，你选不选？"

**实战案例：**

```
计划：优化数据库查询性能

/plan-ceo-review (精选模式):
"基线计划：添加索引，优化慢查询。

扩张机会（可选）：
1. 添加查询缓存（S 努力，中等风险）
2. 引入读写分离（M 努力，高风险）
3. 实现查询分析仪表板（S 努力，低风险）

每个都是独立的。你想要哪个？"
```

**我的经验：** 精选模式是最常用的。它平衡了"聚焦"和"不错过机会"。

#### 模式 3：保持范围（HOLD SCOPE）

**何时使用：** Bug 修复、重构、基础设施改进

**思维模式：** "范围是正确的，让它无懈可击"

**关键问题：**
- 复杂度检查："是否超过 8 个文件？是否引入超过 2 个新类？"
- 最小集："什么可以推迟而不阻碍核心目标？"
- 严谨审查："每个边界情况、每个错误路径、每个测试场景？"

**输出：** 严谨的技术计划（无扩张选项）

**实战案例：**

```
计划：修复支付系统的竞态条件

/plan-ceo-review (保持范围):
"范围是正确的。让我们确保它无懈可击。

审查：
- 数据库锁策略
- 并发测试覆盖
- 回滚方案
- 监控告警
- 灰度发布计划

不添加新功能，只确保现有功能可靠。"
```

**我的经验：** 保持范围模式看起来"不性感"，但它是最常用的模式之一。可靠性比新功能更重要。

#### 模式 4：范围缩减（SCOPE REDUCTION）

**何时使用：** 计划过于复杂、时间紧迫、资源有限

**思维模式：** "什么是最小可交付价值？"

**关键问题：**
- 无情切割："什么是绝对最小的能交付价值的东西？"
- 后续 PR："什么可以放在 follow-up PR？"
- 核心 vs 外围："什么必须一起发布？什么可以分开？"

**输出：** 精简计划 + 推迟列表

**实战案例：**

```
计划：构建完整的电商系统

/plan-ceo-review (缩减模式):
"这个计划太大了。让我们找到最小楔子。

核心（必须现在做）：
- 商品列表
- 购物车
- 结账

推迟（后续 PR）：
- 用户评价
- 推荐系统
- 优惠券
- 多地址配送
- 退款流程

先做核心，验证有人买单。"
```

**我的经验：** 范围缩减是最难的，因为它需要说"不"。但往往是最有价值的。

### 模式选择的智慧

如何决定用哪个模式？

系统有上下文感知：
- 绿地功能 → 默认扩张
- 功能增强 → 默认精选
- Bug 修复 → 默认保持
- 重构 → 默认保持
- 超过 15 个文件 → 建议缩减
- 用户说"做大" → 扩张
- 用户说"简单修复" → 保持

**但你可以覆盖。**

这就是"AI 辅助决策"，不是"AI 代替决策"。

### 实施替代方案：强制性的多角度思考

`/plan-ceo-review`有个强制性设计：**必须生成至少 2 个实现方案**。

这避免了"单一方案偏见"。

#### 方案结构

每个方案包括：
```
方案 A: {名称}
  摘要：[1-2 句话]
  工作量：[S/M/L/XL]
  风险：[低/中/高]
  优点：[2-3 点]
  缺点：[2-3 点]
  复用：[利用了哪些现有代码/模式]
```

#### 方案类型

**最小可行方案：**
- 文件最少
- 差异最小
- 上线最快
- 目的：验证核心假设

**理想架构方案：**
- 长期最优
- 最优雅
- 可扩展
- 目的：指明方向

**创意方案（可选）：**
- 意想不到的方法
- 不同的问题框架
- 目的：激发创新

#### 推荐规则

系统会给出推荐，并说明理由：

```
推荐：选择 [方案 A] 因为 [一行理由，映射到工程偏好]
```

**工程偏好包括：**
- DRY（不要重复自己）
- 充分测试
- 充分工程（不过度也不欠妥）
- 边界情况处理
- 可观测性
- 最小差异

### 核爆炸级范围挑战

在模式选择之前，`/plan-ceo-review`会做"前提挑战"。

这是为了确保计划在正确的轨道上。

#### 0A. 前提挑战

三个问题：
1. "这是正确的问题吗？不同的框架能否产生更简单或更有影响力的解决方案？"
2. "真正的用户/业务结果是什么？计划是达到结果的最直接路径吗？还是解决了代理问题？"
3. "如果我们什么都不做会怎样？真正的痛点还是假设的？"

#### 0B. 现有代码杠杆

关键问题：
1. "有什么现有代码已经部分或完全解决了每个子问题？"
2. "这个计划是否在重建已经存在的东西？如果是，为什么重建比重构更好？"

**这是最常见的浪费：重新发明轮子。**

#### 0C. 梦想状态映射

描述 12 个月后的理想状态：

```
当前状态 → 这个计划 → 12 个月理想
[描述]      [描述增量]    [描述目标]
```

计划是否朝着理想状态前进？

### 时间审问：提前解决实施中的决策

`/plan-ceo-review`会问："实施过程中会做什么决策，应该现在解决？"

它模拟实施过程：
```
第 1 小时（基础）：     实施者需要知道什么？
第 2-3 小时（核心逻辑）：会遇到什么歧义？
第 4-5 小时（集成）：   什么会让他们惊讶？
第 6+ 小时（抛光/测试）：会希望提前计划了什么？
```

**关键设计：** 这些问题现在问，不是"以后再说"。

因为"以后再说"意味着实施者会卡住，或者做出次优决策。

### 持久化：CEO 计划的存储与继承

在扩张和精选模式下，`/plan-ceo-review`会写 CEO 计划文档：

```bash
~/.gstack/projects/$SLUG/ceo-plans/{date}-{feature-slug}.md
```

格式：
```markdown
---
status: ACTIVE
---
# CEO Plan: {功能名称}
生成于 /plan-ceo-review {日期}
分支：{branch} | 模式：{EXPANSION / SELECTIVE EXPANSION}
仓库：{owner/repo}

## 愿景

### 10 倍检查
{10 倍愿景描述}

### 理想状态
{理想状态描述 — 仅扩张模式}

## 范围决策

| # | 提案 | 工作量 | 决策 | 理由 |
|---|------|--------|------|------|
| 1 | {提案} | S/M/L | 接受/推迟/跳过 | {为什么} |

## 接受的范围（添加到计划）
- {项目列表}

## 推迟到 TODOS.md
- {带上下文的项}
```

**为什么重要？**
- 愿景和决策超出对话生命周期
- 下游技能可以读取
- 团队可以 review

### 设计文档推广：从个人到团队

如果愿景产生了引人注目的方向，`/plan-ceo-review`会问：

"这次审查产生了{N}个接受的范围扩张。想把它推广到 repo 的设计文档吗？"

选项：
- A) 推广到 `docs/designs/{FEATURE}.md`（提交到 repo，团队可见）
- B) 保留在 `~/.gstack/projects/`（仅个人参考）
- C) 跳过

**这是个人思考到团队知识的转变。**

每个`/plan-ceo-review`运行后，会显示审查就绪仪表板：

+====================================================================+
|                    审查就绪仪表板                                   |
+====================================================================+
| 审查            | 运行次数 | 最后运行           | 状态      | 必需   |

| Eng Review      |  0   | —                   | —         | YES    |

- 显示每次审查的状态（CLEAR / issues_open）

### 实战案例：从"图片上传"到"智能列表创建"

让我详细讲讲开头的案例。

#### 初始计划

```
创始人："我要加个图片上传功能。用户拍张照，上传到服务器，保存。"

计划：
- 前端：文件选择器
- 后端：接收上传，保存到 S3
- 数据库：存储图片 URL
- 估计：3 天
```

#### /plan-ceo-review 的介入

```
/plan-ceo-review 启动。

读取计划后，它问：

"让我挑战一下前提。

用户真正要的是什么？

是'上传图片'，还是'快速创建一个能卖出去的商品列表'？

如果是后者，整个方案就不同了。"
```

#### 模式选择

```
"基于这是绿地功能，我建议用 SCOPE EXPANSION 模式。

但你可以选择：
A) SCOPE EXPANSION — 构想宏伟版本
B) SELECTIVE EXPANSION — 保持基线 + 精选选项
C) HOLD SCOPE — 让当前计划无懈可击
D) SCOPE REDUCTION — 找到最小可行版本

推荐：A，因为这是新功能，有探索空间。"
```

#### 10 倍检查

```
"让我们想想 10 倍版本。

不是'上传 + 保存'，而是：

1. **图片识别** — 从图片识别产品，自动填充 SKU 或型号
2. **网络丰富** — 搜索网络，自动完善标题、描述、规格
3. **封面建议** — 分析多张图片，建议转化率最高的作为封面
4. **质量检测** — 检测图片是否模糊、昏暗、杂乱，给出改进建议
5. **自动裁剪** — 智能裁剪到最佳构图
6. **背景移除** — 一键移除背景，专业效果

每个都是扩张选项。"
```

#### 范围决策

```
"让我们逐个决定：

1. 图片识别（M 努力，中风险）
   推荐：接受。这是真正的差异化。

2. 网络丰富（S 努力，低风险）
   推荐：接受。30 分钟的工作，巨大价值。

3. 封面建议（L 努力，高风险）
   推荐：推迟。需要 ML 模型，后续 PR。

4. 质量检测（S 努力，低风险）
   推荐：接受。规则基础，简单有效。

5. 自动裁剪（M 努力，中风险）
   推荐：跳过。第三方 API 可解决。

6. 背景移除（S 努力，低风险）
   推荐：接受。有成熟库。"
```

#### 最终方案

```
"范围决策完成。

接受的范围：
- 图片识别（核心差异化）
- 网络丰富（快速价值）
- 质量检测（用户体验）
- 背景移除（专业效果）

推迟：
- 封面建议（需要 ML）

跳过：
- 自动裁剪（第三方解决）

现在，让我们审查技术架构..."
```

#### 结果

最终产品：
- 用户上传一张照片
- AI 识别产品，自动填充 80% 的信息
- 用户只需确认和补充
- 列表创建时间从 15 分钟降到 2 分钟

**这不是"图片上传功能"，是"智能列表创建系统"。**

### 认知模式：伟大 CEO 的思考方式

`/plan-ceo-review`编码了 18 个认知模式。这些不是清单，是思考本能。

#### 1. 分类本能（贝佐斯单向/双向门）
每个决策按可逆性×重要性分类。大多数是双向门——快速行动。

#### 2. 偏执扫描（格罗夫"只有偏执者生存"）
持续扫描战略转折点、文化漂移、人才侵蚀。

#### 3. 逆向反射（芒格"反过来想"）
对每个"如何赢"也问"什么会导致失败"。

#### 4. 专注即减法（乔布斯从 350 产品到 10）
主要价值是*不*做什么。默认：更少事情，更好。

#### 5. 人才优先排序（霍洛维茨"人才、产品、利润"）
人才密度解决大多数其他问题（哈斯廷斯）。

#### 6. 速度校准
快速是默认。只对不可逆 + 高重要性决策放慢。70% 信息就足够决策（贝佐斯）。

#### 7. 代理怀疑
我们的指标还在服务用户吗？还是变成自指的？（贝佐斯 Day 1）

#### 8. 叙事连贯性
艰难决策需要清晰框架。让"为什么"可读，不是每个人都高兴。

#### 9. 时间深度
用 5-10 年弧思考。对重大赌注应用遗憾最小化（贝佐斯）。

#### 10. 创始人模式偏见
深度参与不是微观管理，如果它扩展（而非限制）团队思考（切斯基/格雷厄姆）。

#### 11. 战时意识
正确诊断和平 vs 战时。和平习惯杀死战时公司（霍洛维茨）。

#### 12. 勇气积累
信心*来自*做艰难决策，不是之前。"挣扎就是工作"。

#### 13. 意志作为战略
有意志力。世界屈服于长时间用力推一个方向的人（阿尔特曼）。

#### 14. 杠杆痴迷
找到小努力创造大输出的输入。技术是终极杠杆——一人用正确工具可胜过 100 人无工具（阿尔特曼）。

#### 15. 层级即服务
每个界面决策回答"用户应该先看到什么，第二，第三？"尊重他们的时间，不是美化像素。

#### 16. 边界情况偏执（设计）
如果名字 47 字符怎么办？零结果？网络失败？首次用户 vs 老用户？空状态是功能，不是事后想法。

#### 17. 减法默认
"尽可能少的设计"（拉姆斯）。如果 UI 元素不赚它的像素，砍掉。功能膨胀比缺失功能更快杀死产品。

#### 18. 为信任设计
每个界面决策建立或侵蚀用户信任。像素级关于安全、身份、归属的意图。

**这些认知模式塑造每个推荐。**

### 我的建议：如何最大化/plan-ceo-review 的价值

#### 建议 1：诚实面对模式选择

不要为了"做大"而选扩张模式。

如果这是 bug 修复，就用保持范围。如果时间紧迫，就用缩减模式。

**模式是工具，不是身份。**

#### 建议 2：认真对待替代方案

即使你已经有倾向，也要认真考虑所有方案。

**替代方案不是形式，是保险。**

它防止你掉进"唯一方案陷阱"。

#### 建议 3：保存所有 CEO 计划

即使计划被推翻，也保存它。

6 个月后回看，你会看到思考的演进。

而且，有时候被推翻的计划里藏着被忽略的洞察。

#### 建议 4：让下游参与

如果`/plan-ceo-review`接受了范围扩张，确保`/plan-eng-review`知道。

**上下文传递是 gstack 的核心。**

### 结语：战略思维的可编码化

大多数人认为"战略思维"是不可编码的直觉。

我不同意。

在 AI 时代，执行的边际成本趋近于零。

**战略的价值被放大了 1000 倍。**
