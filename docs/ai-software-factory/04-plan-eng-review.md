# 第四章：/plan-eng-review——像工程经理一样思考架构

## 从"能做"到"可靠"的跨越

有个团队找我："我们的支付系统有 bug，用户被重复收费。"

我看了他们的代码。功能能工作——用户能付款，钱能到账。

但`/plan-eng-review`发现了问题：

- 没有数据库锁，并发请求会重复扣款
- 没有幂等性检查，网络超时重试会重复收费
- 没有失败回滚，扣款成功但订单创建失败时钱不退
- 没有监控告警，问题发生后 3 天才发现

**这不是"有 bug"，是"架构缺陷"。**

在 gstack 里，`/plan-eng-review`的角色就是确保架构可靠。

### 为什么需要工程审查

在传统软件公司，谁负责架构？

**资深工程师/工程经理。**

他们的价值是什么？
- 看到别人看不到的边界情况
- 预见到 6 个月后的问题
- 知道什么可以走捷径，什么必须做对

但他们是稀缺资源：
- 全公司就 2-3 个
- 档期排到 3 个月后
- 同时只能在一个项目上

**/plan-eng-review 把这种能力编码成 AI。**

它不是"更快的工程师"，是"系统化的架构思维"。

### 首要指令：零沉默失败

`/plan-eng-review`有 9 条首要指令。第一条是：

**零沉默失败。**

每个失败模式必须可见——对系统、对团队、对用户。

如果失败能沉默地发生，那是关键缺陷。

#### 实战应用

```
计划：添加支付处理功能

/plan-eng-review:
"让我追踪所有失败模式。

1. API 超时 → 有重试吗？有超时吗？
2. API 返回 429（限流）→ 有退避吗？
3. API 返回畸形 JSON → 有错误处理吗？
4. 数据库连接池耗尽 → 有监控吗？
5. 记录未找到 → 返回 nil 还是抛异常？

每个都要明确处理。"
```

**关键设计：** 不是"处理错误"，是"命名每个异常类，什么触发它，什么捕获它，用户看到什么，是否测试"。

### 错误与救援映射：强制性的表格

`/plan-eng-review`有个强制性设计：**错误与救援表**。

这不是可选项，是必选项。

#### 表格结构

```
方法/代码路径          | 什么会出错               | 异常类
-----------------------|--------------------------|-----------------
ExampleService#call    | API 超时                 | TimeoutError
                       | API 返回 429             | RateLimitError
                       | API 返回畸形 JSON        | JSONParseError
                       | DB 连接池耗尽            | ConnectionPoolExhausted
                       | 记录未找到               | RecordNotFound
-----------------------|--------------------------|-----------------

异常类                 | 救援？ | 救援动作          | 用户看到
-----------------------|--------|-------------------|------------------
TimeoutError           | 是     | 重试 2 次，然后抛出 | "服务暂时不可用"
RateLimitError         | 是     | 退避 + 重试        | 无（透明）
JSONParseError         | 否 ← 缺口 | —               | 500 错误 ← 糟糕
ConnectionPoolExhausted| 否 ← 缺口 | —               | 500 错误 ← 糟糕
RecordNotFound         | 是     | 返回 nil，记录警告  | "未找到"消息
```

#### 规则

1. **捕获所有错误处理是异味**
   - `rescue StandardError`、`catch (Exception e)`、`except Exception` 是异味
   - 命名具体异常

2. **仅通用日志记录不足**
   - 捕获错误时只记录"出错了"不够
   - 记录完整上下文：尝试什么、参数是什么、哪个用户/请求

3. **每个被救援的错误必须**：
   - 带退避重试，或
   - 优雅降级并给用户可见消息，或
   - 带上下文重新抛出

   "吞掉并继续"几乎从不可接受。

4. **LLM/AI 服务调用特别是**：
   - 响应畸形时发生什么？
   - 响应为空时发生什么？
   - 幻觉无效 JSON 时发生什么？
   - 模型返回拒绝时发生什么？

   每个都是不同的失败模式。

#### 缺口检测

任何`RESCUED=N`的行都是**关键缺口**。

```
JSONParseError | 否 ← 缺口

这意味着：如果 API 返回畸形 JSON，你的应用会崩溃，用户看到 500 错误。

这是不可接受的。
```

### 数据流追踪：四条路径

`/plan-eng-review`要求为每个新数据流绘制 ASCII 图。

不是"happy path"，是**四条路径**：

```
输入 → 验证 → 转换 → 持久化 → 输出
  │       │        │         │       │
  ▼       ▼        ▼         ▼       ▼
[nil?] [无效？]  [异常？]   [冲突？] [过时？]
[空？] [太长？]  [超时？]   [重复键？][部分？]
[错误   [错误类型？][OOM？]    [锁定？] [编码？]
 类型？]
```

#### 实战案例

```
计划：添加用户导入功能

/plan-eng-review:
"让我追踪数据流。

Happy Path（数据正确流动）：
用户上传 → 验证 CSV → 解析 → 批量插入 → 确认

Nil Path（输入缺失）：
用户没上传文件 → 验证捕获？→ 显示错误消息

Empty Path（输入存在但为空）：
用户上传空文件 → 解析捕获？→ 显示"文件为空"

Error Path（上游失败）：
DB 连接失败 → 插入捕获？→ 重试？回滚？

每个路径都要明确处理。"
```

**关键设计：** 为每个节点问：什么发生时？是否测试？

### 交互边界情况：用户可见的失败

数据流是后端。交互是前端。

`/plan-eng-review`要求为每个新用户可见交互评估：

```
交互                | 边界情况              | 处理？ | 如何？
--------------------|----------------------|--------|--------
表单提交            | 双击提交             | ?      |
                    | CSRF token 过时提交   | ?      |
                    | 部署期间提交         | ?      |
异步操作            | 用户中途导航离开     | ?      |
                    | 操作超时             | ?      |
                    | 进行中时重试         | ?      |
列表/表格视图       | 零结果               | ?      |
                    | 10,000 结果          | ?      |
                    | 页面中间结果变化     | ?      |
后台任务            | 10 项中处理 3 项后失败 | ?      |
                    | 任务运行两次（重复）  | ?      |
                    | 队列积压 2 小时       | ?      |
```

**任何未处理的边界情况都是缺口。**

### 图表的强制力量

`/plan-eng-review`最强大的设计之一：**强制图表**。

不是"最好有"，是"必须有"。

#### 必需的 ASCII 图

1. **系统架构** — 显示新组件及其与现有组件的关系
2. **数据流**（包括阴影路径）— 四条路径都绘制
3. **状态机** — 每个有状态对象，包括不可能/无效转换
4. **错误流** — 错误如何传播
5. **部署序列** — 部署步骤
6. **回滚流程** — 出问题时如何回滚

#### 为什么图表如此强大

**图表迫使隐藏假设暴露。**

你可以用文字说"系统处理错误"。

但当你绘制错误流时，你必须具体：
- 什么组件捕获？
- 什么日志记录？
- 什么通知用户？
- 什么触发告警？

**模糊是架构之敌。**

### 测试审查：从"有测试"到"有信心的测试"

`/plan-eng-review`对测试的审查是系统的。

不是"有测试吗？"，是"**什么测试能让你有信心在周五凌晨 2 点发布？**"

#### 测试金字塔审查

```
新 UX 流程：
  [列出每个新用户可见交互]

新数据流：
  [列出每个新路径]

新代码路径：
  [列出每个新分支、条件、执行路径]

新后台任务/异步工作：
  [列出每个]

新集成/外部调用：
  [列出每个]

新错误/救援路径：
  [列出每个 — 交叉引用第 2 节]
```

对每个项问：
- 什么类型测试覆盖？（单元/集成/系统/E2E）
- 计划中有测试吗？如果没有，写测试规范头
- happy path 测试是什么？
- failure path 测试是什么？（具体 — 哪个失败？）
- 边界情况测试是什么？（nil、空、边界值、并发访问）

#### 测试雄心检查

```
什么测试能让你有信心在周五凌晨 2 点发布？
什么测试敌对 QA 工程师会写来破坏这个？
什么是混沌测试？
```

**这些不是修辞问题。是必须回答的问题。**

### 可观测性审查：新系统会失败

`/plan-eng-review`有个基本假设：**新系统会失败。**

问题不是"是否会失败"，是"失败时你能看到吗？"

#### 评估项

- **日志记录** — 每个新代码路径：入口、出口、每个重要分支的结构化日志行？
- **指标** — 每个新功能：什么指标告诉你它在工作？什么告诉你它坏了？
- **追踪** — 新跨服务/跨工作流：追踪 ID 传播？
- **告警** — 应该存在什么新告警？
- **仪表板** — 第 1 天想要什么新仪表板面板？
- **可调试性** — 如果 3 周后报告 bug，你能仅从日志重建发生了什么吗？
- **管理工具** — 需要新管理 UI 或 rake 任务？
- **操作手册** — 每个新失败模式：操作响应是什么？

#### 扩张和精选模式的额外问题

```
什么可观测性会让这个功能*乐于*操作？
```

**不是"能操作"，是"乐于操作"。**

### 部署与回滚审查：部署不是原子的

`/plan-eng-review`有个关键洞察：**部署不是原子的**。

旧代码和新代码同时运行。这创造风险。

#### 评估项

- **迁移安全** — 每个新 DB 迁移：向后兼容？零停机？表锁定？
- **功能标志** — 任何部分应该在功能标志后？
- **部署顺序** — 正确顺序：先迁移，再部署？
- **回滚计划** — 明确步骤。
- **部署时风险窗口** — 旧代码和新代码同时运行 — 什么会坏？
- **环境一致性** — 在 staging 测试？
- **部署后验证清单** — 前 5 分钟？前 1 小时？
- **冒烟测试** — 部署后应该立即运行什么自动检查？

#### 实战案例

```
计划：添加新的支付网关

/plan-eng-review:
"部署审查：

1. 迁移安全：添加新表，向后兼容 ✓
2. 功能标志：应该在 feature_flag(:new_gateway) 后 ✓
3. 部署顺序：先迁移，再部署代码 ✓
4. 回滚计划：
   - 关闭功能标志
   - 回滚代码
   - 删除新表（如果无数据）
5. 部署时风险：旧代码不知道新表，但不会访问 ✓
6. 环境一致性：已在 staging 测试 ✓
7. 部署后验证：
   - 前 5 分钟：监控错误率
   - 前 1 小时：第一笔真实交易成功
8. 冒烟测试：
   - 创建测试订单
   - 完成支付
   - 验证订单状态"
```

### 长期轨迹审查：1 年后的视角

`/plan-eng-review`会问："12 个月后读这个计划，明显吗？"

这是**时间旅行测试**。

#### 评估项

- **技术债务引入** — 代码债务、操作债务、测试债务、文档债务
- **路径依赖** — 这使未来变更更难吗？
- **知识集中** — 文档足够新工程师理解？
- **可逆性** — 评分 1-5：1=单行道，5=容易可逆
- **生态系统匹配** — 与 Rails/JS 生态系统方向一致？
- **1 年问题** — 12 个月后的新工程师读这个计划 — 明显吗？

#### 扩张和精选模式的额外问题

```
这个部署后是什么？Phase 2？Phase 3？
架构支持那个轨迹吗？
平台潜力：这创造其他功能可利用的能力吗？
```

### 设计审查：CEO 调用设计师

如果计划涉及 UI，`/plan-eng-review`会包括设计审查。

不是像素级审查——那是`/plan-design-review`。是**设计意图审查**。

#### 评估项

- **信息架构** — 用户先看到什么，第二，第三？
- **交互状态覆盖图**：
  ```
  功能 | 加载中 | 空 | 错误 | 成功 | 部分
  ```
- **用户旅程连贯性** — 绘制情感弧
- **AI 垃圾风险** — 计划描述通用 UI 模式？
- **DESIGN.md 对齐** — 计划匹配声明的设计系统？
- **响应式意图** — 移动端提到还是事后想法？
- **可访问性基础** — 键盘导航、屏幕阅读器、对比度、触摸目标

#### 必需的 ASCII 图

**用户流** — 显示屏幕/状态和转换。

### 与下游的协同：测试计划传递

`/plan-eng-review`完成时，写测试计划工件到`~/.gstack/projects/`。

当后来运行`/qa`时，它自动拾取那个测试计划。

**这就是工件驱动的上下文传递。**

没有手动复制粘贴。没有"我记得测试计划说..."。

### 实战案例：支付系统的架构审查

让我详细讲讲开头的案例。

#### 初始计划

```
计划：修复支付系统的重复收费 bug

当前实现：
- 用户点击"支付"
- 前端调用后端 API
- 后端调用 Stripe
- Stripe 扣款
- 后端创建订单
- 返回成功
```

#### /plan-eng-review 的审查

```
/plan-eng-review 启动。

读取计划后，它说：

"让我先做系统审查。

当前状态：
- 支付系统处理每月$100 万交易
- 3 个工程师维护
- 无专职 on-call
- 上次事故：2 周前，重复收费$5 万

什么已经变化：
- 过去 30 天 12 个提交
- 2 个 FIXME 评论在 payment_service.rb
- 1 个 TODO 评论关于幂等性

已知痛点（来自 TODOS.md）：
- TODO: 添加幂等性检查到支付
- TODO: 添加支付失败重试队列
- FIXME: 并发支付可能导致重复收费 ← 就是这个 bug"
```

#### 错误与救援映射

```
/plan-eng-review:
"让我做错误与救援映射。

方法/代码路径          | 什么会出错               | 异常类
-----------------------|--------------------------|-----------------
PaymentService#charge  | Stripe API 超时          | TimeoutError
                       | Stripe 返回错误          | StripeError
                       | DB 插入失败              | ActiveRecord::StatementInvalid
                       | 并发重复收费            | RaceConditionError
                       | 网络失败                | NetworkError
-----------------------|--------------------------|-----------------

异常类                 | 救援？ | 救援动作          | 用户看到
-----------------------|--------|-------------------|------------------
TimeoutError           | 否 ← 缺口 | —               | 500 错误 ← 糟糕
StripeError            | 是     | 记录错误，显示消息 | "支付失败，请重试"
ActiveRecord::...      | 否 ← 缺口 | —               | 500 错误 ← 糟糕
RaceConditionError     | 否 ← 缺口 | —               | 500 错误 ← 糟糕
NetworkError           | 否 ← 缺口 | —               | 500 错误 ← 糟糕

4 个关键缺口。这是不可接受的。"
```

#### 数据流追踪

```
/plan-eng-review:
"让我追踪数据流。

Happy Path：
用户点击 → 前端调用 API → 后端验证 → 调用 Stripe → 
扣款成功 → 创建订单 → 返回成功

Nil Path（用户快速点击两次）：
第一次点击 → API 调用 → Stripe 扣款 → 
第二次点击（同时）→ 第二个 API 调用 → 第二个 Stripe 扣款 ← 重复收费！

Empty Path（Stripe 返回空响应）：
调用 Stripe → 返回空 → 代码假设成功 → 创建订单 → 
用户被收费但订单不存在 ← 数据不一致！

Error Path（DB 失败）：
Stripe 扣款成功 → DB 插入失败 → 
钱扣了但订单没创建 ← 钱丢了！

3 个关键缺口。每个都需要处理。"
```

#### 架构方案

```
/plan-eng-review:
"让我生成实现方案。

方案 A：幂等性键（推荐）
  摘要：为每个支付生成唯一幂等性键，Stripe 保证不重复扣款
  工作量：M
  风险：低
  优点：
    - Stripe 原生支持
    - 简单可靠
    - 客户端生成键，服务端验证
  缺点：
    - 需要客户端变更
  复用：现有 Stripe 集成

方案 B：数据库锁
```
