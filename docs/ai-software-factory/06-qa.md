# 第六章：/qa——系统性测试与自动修复

## 从"手动测试"到"系统测试"的转变

传统 QA 是什么？
- 手动点击每个按钮
- 填写表单看是否工作
- 检查页面是否加载
- 报告发现的 bug

这在小项目中可行。但在大项目中？
- 100 个页面
- 50 个用户流程
- 10 种设备尺寸
- 无数边界情况

**手动测试无法规模化。**

在 gstack 里，`/qa`不是"更快的手动测试"，而是**系统性测试方法论**。

### 四层测试模式

`/qa`提供四种模式，适应不同场景：

#### 快速模式（--quick）

**30 秒烟雾测试。**

- 访问首页 + 前 5 个导航目标
- 检查：页面加载？控制台错误？断链？
- 生成健康分数
- **无详细问题文档**

**适用场景：**
- 快速验证部署
- 日常冒烟测试
- 紧急修复验证

#### 标准模式（默认）

**5-15 分钟系统探索。**

- 系统性访问每个可到达页面
- 每页执行完整检查清单
- 文档化 5-10 个有证据的问题
- 生成详细健康分数

**适用场景：**
- 功能开发完成后的验证
- PR 合并前的测试
- 发布前的全面测试

#### 穷尽模式（--exhaustive）

**深度测试所有问题。**

- 包含标准模式的所有内容
- 额外测试低优先级/外观问题
- 更深入的交互测试
- 更全面的响应式测试

**适用场景：**
- 关键功能发布
- 用户体验重点优化
- 设计系统一致性验证

#### 回归模式（--regression baseline.json）

**变更影响分析。**

- 运行标准模式
- 加载之前的基准文件
- 比较：哪些问题修复了？哪些是新的？
- 显示分数差异和回归报告

**适用场景：**
- 迭代开发中的质量追踪
- 修复验证
- 趋势分析

### Diff-aware 模式：智能测试范围

`/qa`最强大的设计是**Diff-aware 模式**。

当你在功能分支上运行`/qa`而没有指定 URL 时：

```
你：/qa

/qa：检测到你在 feature/new-payment 分支
     分析 git diff main...HEAD
     识别受影响的页面/路由：
     - /checkout (修改了 checkout_controller.rb)
     - /profile (修改了 user_profile.vue)
     - /api/payments (新增了 payments_controller.rb)
     
     检测到本地应用在 :3000
     开始测试受影响的页面...
```

**这不是猜测，是精确分析。**

#### 如何工作

1. **分析分支差异**
   ```bash
   git diff main...HEAD --name-only
   git log main..HEAD --oneline
   ```

2. **识别受影响页面/路由**
   - 控制器/路由文件 → URL 路径
   - 视图/模板/组件文件 → 渲染页面
   - 模型/服务文件 → 使用这些模型的控制器
   - CSS/样式文件 → 包含样式的页面
   - API 端点 → 直接测试 API

3. **检测运行应用**
   - 检查常见本地端口 (:3000, :4000, :8080)
   - 如果找不到，检查 staging/preview URL

4. **针对性测试**
   - 只测试受影响的页面
   - 验证变更意图（从提交消息和 PR 描述）
   - 检查相邻页面是否有回归

**结果：** 不是盲目测试整个应用，而是**精准测试变更影响**。

### 自动修复循环：Test → Fix → Verify

`/qa`不只是发现问题，还**自动修复**。

#### 修复循环

1. **定位源码**
   - Grep 错误消息、组件名称、路由定义
   - Glob 文件模式匹配受影响页面

2. **修复**
   - 阅读源码，理解上下文
   - 做**最小修复** — 解决问题的最小变更
   - 不重构周围代码，不"改进"无关事物

3. **提交**
   ```bash
   git add <only-changed-files>
   git commit -m "fix(qa): ISSUE-NNN — short description"
   ```
   - 每个修复一个提交
   - 从不捆绑多个修复

4. **重新测试**
   - 导航回受影响页面
   - 截取**前后对比截图**
   - 检查控制台错误
   - 使用 `snapshot -D` 验证变化

5. **分类**
   - **verified**：重新测试确认修复有效
   - **best-effort**：修复但无法完全验证
   - **reverted**：检测到回归 → `git revert HEAD`

#### 自动回归测试

对于 verified 修复，`/qa`自动生成回归测试：

1. **研究项目测试模式**
   - 读取 2-3 个现有测试文件
   - 匹配：文件命名、导入、断言风格、setup/teardown 模式

2. **追溯 bug 代码路径**
   - 什么输入/状态触发了 bug？
   - 代码路径经过哪些分支/函数调用？
   - 在哪里失败？
   - 其他输入能否触发相同代码路径？

3. **编写回归测试**
   - 设置触发 bug 的前提条件
   - 执行暴露 bug 的操作
   - 断言正确行为（不是"它渲染"或"不抛异常"）
   - 如果找到相邻边界情况，也测试它们
   - 包含完整归属注释：
     ```javascript
     // Regression: ISSUE-NNN — {what broke}
     // Found by /qa on {YYYY-MM-DD}
     // Report: .gstack/qa-reports/qa-report-{domain}-{date}.md
     ```

4. **运行和验证**
   - 只运行新测试文件
   - 通过 → 提交：`git commit -m "test(qa): regression test for ISSUE-NNN"`
   - 失败 → 修复测试一次，仍失败 → 删除测试

**关键设计：** 回归测试必须看起来像同一开发者写的。

### 健康分数：量化质量

`/qa`计算**健康分数**，而不是主观判断。

#### 评分规则

**控制台（权重：15%）**
- 0 错误 → 100
- 1-3 错误 → 70  
- 4-10 错误 → 40
- 10+ 错误 → 10

**链接（权重：10%）**
- 0 断链 → 100
- 每个断链 → -15（最低 0）

**按类别评分（视觉、功能、UX、内容、性能、可访问性）**
每个类别从 100 开始。每个发现扣分：
- Critical 问题 → -25
- High 问题 → -15  
- Medium 问题 → -8
- Low 问题 → -3
最低 0 每类别。

#### 最终分数

`score = Σ (category_score × weight)`

**为什么重要？**
- 客观比较不同版本
- 量化改进效果
- 设置质量门禁

### WTF-likelihood：自我调节启发式

`/qa`有个内置的**风险控制系统**。

每 5 个修复（或任何回退后），计算 WTF-likelihood：

```
WTF-LIKELIHOOD:
  起始 0%
  每次回退：+15%
  每个触及 >3 文件的修复：+5%
  第 15 个修复后：每个额外修复 +1%
  所有剩余 Low 严重性：+10%
  触及无关文件：+20%
```

**如果 WTF > 20%：立即停止。** 显示已做工作，询问是否继续。

**硬上限：50 个修复。** 之后停止，无论剩余问题。

**为什么重要？**
- 防止过度修复
- 避免引入新问题
- 保持修复质量

### 框架特定指导

`/qa`了解不同框架的常见问题：

#### Next.js
- 检查水合错误（`Hydration failed`、`Text content did not match`）
- 监控 `_next/data` 请求（404 表示数据获取中断）
- 测试客户端导航（不仅是 `goto`）
- 检查 CLS（累积布局偏移）

#### Rails
- 检查 N+1 查询警告（开发模式）
- 验证 CSRF token 存在
- 测试 Turbo/Stimulus 集成
- 检查 flash 消息正确显示/消失

#### WordPress
- 检查插件冲突（不同插件的 JS 错误）
- 验证管理员栏可见性
- 测试 REST API 端点（`/wp-json/`）
- 检查混合内容警告

#### 通用 SPA（React、Vue、Angular）
- 使用 `snapshot -i` 导航（`links` 命令错过客户端路由）
- 检查陈旧状态（导航离开再返回）
- 测试浏览器后退/前进
- 监控内存泄漏

### 实战案例：电商结账流程测试

让我详细讲个案例。

#### 初始状态

团队开发了新结账流程：
- 新的结账页面
- 集成 Stripe 支付
- 订单确认邮件
- 库存更新

他们运行了单元测试，全部通过。

#### /qa 的介入

```
你：/qa

/qa：检测到你在 checkout-redesign 分支
     分析 git diff：
     - app/views/checkout/index.html.erb (新增)
     - app/controllers/checkout_controller.rb (新增)  
     - app/services/payment_service.rb (修改)
     - app/mailers/order_mailer.rb (修改)
     - app/models/inventory.rb (修改)
     
     识别受影响页面：
     - /checkout
     - /orders/confirmation
     
     检测到本地应用在 :3000
     开始测试...
```

#### 发现的问题

**Critical 问题：**
1. **结账表单提交空必填字段**
   - 截图：表单提交前
   - 截图：提交后（无错误消息，页面刷新）
   - 控制台：无 JS 错误
   - 问题：前端验证缺失，后端未返回错误

2. **移动导航菜单不关闭**
   - 截图：移动视图，菜单打开
   - 截图：选择项目后，菜单仍打开遮挡内容
   - 问题：缺少关闭逻辑

**High 问题：**
3. **仪表板图表在 1024px 以下重叠侧边栏**
   - 截图：1023px 宽度，图表重叠
   - 问题：响应式断点缺失

4. **控制台错误：Stripe 元素未正确销毁**
   - 控制台：`Stripe elements not properly destroyed`
   - 问题：内存泄漏风险

#### 自动修复

```

- 定位：app/views/checkout/index.html.erb
- 添加前端验证：必填字段检查

- 添加点击处理程序：选择项目后关闭菜单
- 提交：fix(qa): ISSUE-002 — close mobile menu after selection

修复 3：响应式断点
- 定位：app/assets/stylesheets/dashboard.scss
- 添加媒体查询：@media (max-width: 1024px)
- 调整图表宽度
- 提交：fix(qa): ISSUE-003 — fix dashboard chart overlap on mobile

修复 4：Stripe 元素清理
- 定位：app/assets/javascripts/stripe-elements.js
- 添加组件卸载清理
- 提交：fix(qa): ISSUE-004 — properly destroy stripe elements on unmount
```

### 我的建议：如何最大化/qa 的价值

#### 建议 1：使用 Diff-aware 模式

不要总是指定 URL。

**在功能分支上直接运行 `/qa`。**

让系统智能识别测试范围。

#### 建议 2：信任自动修复

即使你觉得"这个简单，我自己修"。

**让 `/qa` 修复并生成回归测试。**

这确保问题不会再次出现。

#### 建议 3：监控健康分数趋势

不要只看单次数值。

**跟踪健康分数随时间的变化。**

设置质量门禁：分数低于 X 不允许合并。

#### 建议 4：尊重 WTF-likelihood

当 `/qa` 停止时，认真考虑。

**这不是 bug，是保护机制。**

如果需要更多修复，分批进行。

### 结语：QA 的新时代

传统 QA 是成本中心：
- 需要专门的 QA 团队
- 测试是瓶颈
- Bug 发现在后期
- 修复成本高

`/qa` 使 QA 成为价值中心：
- 内置到开发流程
- 即时反馈
- 自动修复
- 预防回归

**这不是"自动化测试"，是"系统性质量保证"。**

在 AI 时代，质量的边际成本趋近于零。

**高质量成为默认，而不是奢侈品。**

所以，投资质量。

运行 `/qa`。
