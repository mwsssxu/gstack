# 第二章：/office-hours——像 YC 合伙人一样思考产品

## 一个价值百万的重新定位

2025 年秋天，有个创始人来找我。他说："Garry，我想做一个每日简报应用，帮我管理多个 Google 日历。"

听起来是个合理的需求，对吧？

然后我问了他几个问题：
- "你为什么要做这个？"
- "你试过什么解决方案？"
- "谁会为这个付钱？"
- "最大的痛点是什么？"

他的回答让我意识到：他真正要做的不是"日历管理应用"，而是"个人首席参谋 AI"。

六个月后，他的产品成了市场上最火的 AI 助手之一，估值翻了 10 倍。

**差别在哪里？在于他问对了问题。**

在 gstack 里，我把这个提问过程编码成了第一个角色：`/office-hours`。

### 为什么 80% 的产品注定失败

让我说个残酷的事实：

**大多数产品失败，不是因为技术不行，是因为它们解决了错误的问题。**

我见过太多这样的案例：

- 花 6 个月做一个功能，上线后发现没人用
- 融了 500 万，做了一个"完整的产品"，结果用户只想要其中一个 10% 的功能
- 团队有 20 个工程师，产品却打不过 3 个人的竞品

为什么？

因为他们都在**用战术的勤奋掩盖战略的懒惰**。

他们不愿意花时间去问那些困难的问题：
- "用户真的需要这个吗？"
- "为什么是现在？"
- "为什么是我们？"
- "如果失败了，最可能的原因是什么？"

在 YC，我们有个传统：每个创始人都要经历"Office Hours"——和 YC 合伙人一对一讨论产品。

这不是"给建议"，是"问问题"。

好的问题比好的答案更有价值。

### 将 YC 式提问编码成 AI

在 gstack 里，我想让每个开发者都能享受到 YC 级别的產品思考。

所以我把 YC 合伙人的提问模式编码成了 AI 技能。

让我给你展示一下`/office-hours`是如何工作的。

#### 第一步：识别用户类型

当你运行`/office-hours`时，它不会直接开始给建议。

它会先问你："你的目标是什么？"

然后给你 6 个选项：
1. 正在创业（或考虑创业）
2. 内部创业——公司内的项目
3. 黑客马拉松——有时间限制，需要让人印象深刻
4. 开源/研究——为社区建设或探索想法
5. 学习——自学编程，提升技能
6. 玩得开心——副业，创意出口

**这个分类很重要。**

给创业者的建议和对业余项目爱好者的建议是完全不同的。

创业者需要听到残酷的真相，而爱好者需要的是鼓励和创意碰撞。

#### 第二步：六种强制性问题（创业者模式）

如果你选择"创业"，`/office-hours`会切换到"创业者模式"，问你 6 个强制性问题。

这些问题是精心设计的，每个都直指产品成功的关键。

##### 问题 1：需求现实检验

**问题：** "你有什么最强有力的证据证明有人真的想要这个——不是'感兴趣'，不是'注册了等待列表'，而是如果它消失了会真正沮丧的人？"

**为什么要问这个？**

因为太多创始人把"兴趣"当成"需求"。

- "我们有 500 个等待列表注册！"→ 注册是免费的，不代表需求
- "VC 对这个领域很兴奋！"→ VC 的兴趣不等于用户需求
- "人们说这个想法很有趣！"→ 礼貌的赞美不等于付费意愿

**你想听到的证据：**
- 有人在付费
- 有人在扩大使用
- 有人在你服务中断时打电话给你
- 有人围绕你的产品构建他们的工作流程

**实战案例：**

```
创始人："我们有 1000 个等待列表注册！"
YC 合伙人："多少人付钱了？"
创始人："嗯...我们还没开始收费..."
YC 合伙人："那你没有任何需求证据。"
```

##### 问题 2：现状分析

**问题：** "你的用户现在在用什么方法解决这个问题——即使很糟糕？这个现状的成本是什么？"

**为什么要问这个？**

因为**你的真正竞争对手不是其他创业公司，是"什么都不做"**。

如果用户现在没有用任何方法解决问题，那通常意味着这个问题不够痛。

**你想听到的：**
- 具体的工作流程
- 花费的时间
- 浪费的金钱
- 拼凑在一起的工具
- 专门雇来做这件事的人

**实战案例：**

```
创始人："市场上没有解决方案，这就是机会所在！"
YC 合伙人："那用户现在怎么办？"
创始人："他们...什么都不做？"
YC 合伙人："那就对了。不是机会，是根本没问题。"
```

##### 问题 3：绝望的具体性

**问题：** "命名最需要你的产品的那个具体的人。他是什么职位？什么能让他升职？什么能让他被解雇？什么让他夜不能寐？"

**为什么要问这个？**

因为**类别不是客户，人是**。

"医疗保健企业"不是客户。"Sarah，50 人物流公司的运营经理"才是客户。

你不能用电子邮件发给一个类别。

**你想听到的：**
- 具体的名字
- 具体的职位
- 具体的后果
- 直接从那个人嘴里听到的话

**实战案例：**

```
创始人："我们的目标客户是中小企业。"
YC 合伙人："具体哪个中小企业？"
创始人："就是...需要我们的产品的企业？"
YC 合伙人："说一个名字。"
创始人："呃..."
YC 合伙人："如果你不能说出具体的名字，说明你没跟任何客户聊过。"
```

##### 问题 4：最窄楔子

**问题：** "有人愿意为此付钱的尽可能小的版本是什么——这周，不是等你构建完整个平台？"

**为什么要问这个？**

因为**最小可行产品（MVP）不是"功能最少的产品"，是"能收到钱的最简单东西"**。

太多创始人说："我们需要先构建完整平台，用户才能用。"

错。你应该能在几天内做出有人愿意付钱的东西。

**你想听到的：**
- 一个功能
- 一个工作流程
- 简单到可以在几天内交付
- 有人愿意现在就付钱

**实战案例：**

```
创始人："我们需要构建完整的平台，包括用户认证、支付系统、数据分析..."
YC 合伙人："你能做什么最简单的版本，这周就有人付钱？"
创始人："嗯...其实我们可以用邮件手动发送报告。"
YC 合伙人："那就做那个。"
```

##### 问题 5：观察与意外

**问题：** "你真的坐下来看过有人使用你的产品吗——不帮助他们？他们做了什么让你惊讶的事？"

**为什么要问这个？**

因为**用户的真实行为会推翻你的所有假设**。

如果你没有被用户的行为惊讶过，说明你没在观察。

**你想听到的：**
- 具体的意外行为
- 用户做了产品没设计的事
- 用户绕过了你精心设计的流程
- 用户用你的产品做了你没想到的事

**实战案例：**

```
创始人："用户都很喜欢我们的功能！"
YC 合伙人："你看过他们怎么用吗？"
创始人："我们发了调查问卷..."
YC 合伙人："不。我是说，你坐在他们旁边，看他们用吗？"
创始人："..."
YC 合伙人："那你什么都不知道。"
```

##### 问题 6：未来适配

**问题：** "如果 3 年后世界明显不同了——它肯定会——你的产品会变得更重要还是更不重要？"

**为什么要问这个？**

因为**没有长期愿景的产品，最终会被有愿景的产品打败**。

"AI 会变得越来越好，所以我们会越来越好"——这不是愿景，这是搭便车。

**你想听到的：**
- 具体的关于用户世界如何变化的断言
- 为什么这个变化让你的产品更有价值
- 不是"市场每年增长 20%"这种废话

**实战案例：**

```
创始人："AI 会让一切都变得更好！"
YC 合伙人："具体到你的产品呢？"
创始人："嗯...AI 越好，用我们产品的人越多？"
YC 合伙人："那你的竞争对手也能用 AI。你的独特价值是什么？"
创始人："..."
```

#### 第三步：前提挑战

问完 6 个问题后，`/office-hours`会总结你的回答，然后挑战你的前提。

它会问：
1. "这是正确的问题吗？"
2. "如果我们什么都不做会怎样？"
3. "现有代码中有什么已经部分解决了这个问题？"
4. （创业者模式）"综合诊断证据，它支持这个方向吗？"

然后输出前提声明，让你确认：

```
前提：
1. [声明] — 同意/不同意？
2. [声明] — 同意/不同意？
3. [声明] — 同意/不同意？
```

如果你不同意某个前提，它会修正理解并回溯。

#### 第四步：生成实现方案

这是很多人意想不到的。

`/office-hours`**不会只给一个建议**。

它会生成 2-3 个不同的实现方案，每个都有：
- 名称
- 摘要（1-2 句话）
- 工作量（S/M/L/XL）
- 风险（低/中/高）
- 优点（2-3 点）
- 缺点（2-3 点）
- 复用（利用了哪些现有代码/模式）

规则：
- 至少 2 个方案，最好是 3 个
- 必须有一个是"最小可行"方案（文件最少，差异最小，上线最快）
- 必须有一个是"理想架构"方案（长期最优，最优雅）
- 可以有一个是"创意/横向"方案（意想不到的方法）

然后给出推荐：

```
推荐：选择 [X] 因为 [一行理由]
```

#### 第五步：写设计文档

一旦你批准了方案，`/office-hours`会写一个完整的设计文档。

这个文档会保存到 `~/.gstack/projects/{slug}/{user}-{branch}-design-{datetime}.md`

**关键设计：** 这个文档不是"写完就结束"，它是下游技能的输入。

`/plan-ceo-review`会读它，`/plan-eng-review`会读它，后续的 QA 也会参考它。

**这就是工件驱动的上下文传递。**

### 实战案例：从"日历应用"到"首席参谋 AI"

让我详细讲讲开头那个案例。

#### 初次对话

```
创始人："我想做一个每日简报应用，帮我管理多个 Google 日历。"

/office-hours 启动。

AI："在开始之前，你的目标是什么？"

创始人："正在创业。"

AI："好的。让我先了解一下你的项目。"
    [阅读 CLAUDE.md、TODOS.md、git log]
    "你处于什么阶段？"

创始人："还没有用户，只有一个想法。"
```

#### 六个问题的对话

**问题 1：需求现实**

```
AI："你有什么最强有力的证据证明有人真的想要这个？"

创始人："嗯...我自己就需要！我有 3 个 Google 日历，经常错过会议。"

AI："那是你的需求。其他人呢？"

创始人："我问过几个朋友，他们也有同样的问题。"

AI："他们付钱了吗？"

创始人："还没有产品...但他们说如果有愿意付钱。"

AI："那不是证据。那是假设。"
```

**问题 2：现状**

```
AI："你的用户现在在用什么方法解决这个问题？"

创始人："他们用 Google 日历的提醒功能，还有手动检查。"

AI："成本是什么？"

创始人："他们说每天要花 30 分钟检查日历，还经常错过重要的事。"

AI："具体什么重要的事？"

创始人："比如董事会会议、投资人电话、重要客户会议。"

AI："错过一次的成本是多少？"

创始人："可能是几十万，甚至上百万。"

AI："好。那是真正的痛点。"
```

**问题 3：具体的人**

```
AI："命名最需要你的产品的那个具体的人。"

创始人："嗯...我自己？"

AI："你是创始人，不算。你的第一个付费客户会是谁？"

创始人："可能是...其他创业公司的 CEO？"

AI："具体名字。"

创始人："John，他经营一家 fintech 创业公司，有 5 个日历，经常错过投资人会议。"

AI："好。John 的职位是什么？什么能让他升职？什么能让他被解雇？"

创始人："他是 CEO。如果公司融资成功，他就'升职'了。如果错过投资人会议导致融资失败，他可能被董事会解雇。"

AI："所以他夜不能寐的是融资。你的产品如何帮他？"

创始人："确保他不会错过任何投资人会议，帮他准备会议。"
```

**问题 4：最窄楔子**

```
AI："你能做什么最简单的版本，这周就有人付钱？"

创始人："嗯...我可以手动帮他准备每日简报？"

AI："具体包括什么？"

创始人："每天早上发邮件，列出今天的会议，每个会议的背景信息，需要准备什么。"

AI："那需要多久？"

创始人："大概 30 分钟。"

AI："你能把这个自动化吗？"

创始人："可以写个脚本，从 Google 日历读取会议，然后从 LinkedIn 和 Crunchbase 抓取参会者信息。"

AI："好。那就是你的 MVP。不是完整的平台，是一个脚本 + 手动服务。"
```

**问题 5：观察与意外**

```
AI："你看过别人怎么用日历管理工具吗？"

创始人："我看过几个朋友用...他们经常忽略提醒，因为太多假阳性了。"

AI："有什么让你惊讶的？"

创始人："有个朋友会把所有会议都设置成 15 分钟，即使实际是 1 小时。他说这样能逼自己高效。"

AI："那意味着什么？"

创始人："意味着提醒功能本身不够，需要智能判断哪些会议真的不能错过。"
```

**问题 6：未来适配**

```
AI："3 年后，世界会有什么不同？你的产品会更重要还是更不重要？"

创始人："AI 会越来越好，人们的时间会更值钱..."

AI："具体到你的产品？"

创始人："如果 AI 能自动准备会议简报，那我的产品会更有价值，因为...等等，那我不就被取代了？"

AI："好问题。那你的长期价值是什么？"

创始人："可能是...不仅仅是会议简报，而是整个关系管理？CRM for personal relationships？"

AI："接近了。再想想。"
```

#### 重新定位

问完问题后，`/office-hours`做了个惊人的重新定位：

```
我要挑战你的框架。

你说"每日简报应用"。但你实际描述的是"个人首席参谋 AI"。

从你的回答中，我提取了 5 个能力：

1. **监控你的日历** — 跨所有账户，检测过时信息、缺失地点、权限缺口
2. **生成真正的准备工作** — 不是物流摘要，而是*智力工作*：准备董事会会议、播客、融资会议
3. **管理你的 CRM** — 你要见谁，关系如何，他们想要什么，历史如何
4. **优先排序你的时间** — 主动标记何时需要提前准备，按重要性排序事件
5. **用金钱换杠杆** — 主动寻找可以委托或自动化的事情

建议：不要构建"日历管理应用"。构建"个人首席参谋 AI"。

最窄楔子：每日简报，但不仅列出会议，还告诉你：
- 每个会议的背景
- 参会者的关系历史
- 需要准备什么
- 如果错过会怎样

这样，你不是在做一个"日历工具"，你在做一个"让创始人效率提升 10 倍的 AI"。
```

### 两种模式：创业者 vs 建造者

`/office-hours`有个重要设计：**模式切换**。

不是所有人都需要听到残酷的真相。

#### 创业者模式

当你选择"创业"或"内部创业"时，`/office-hours`是严格的、挑战性的。

- 问 6 个强制性问题
- 挑战你的前提
- 要求具体证据
- 推动你找到最窄楔子

**目标：验证需求，找到产品市场契合点。**

#### 建造者模式

当你选择"黑客马拉松"、"开源"、"学习"或"玩得开心"时，`/office-hours`是热情的、创意的。

它会问：
- "什么版本最酷？什么会让人说'哇'？"
- "你会展示给谁看？什么能让他们印象深刻？"
- "最快能做出什么可以分享的东西？"
- "有什么现有的东西最接近你的想法，你的有什么不同？"
- "如果你有无限时间，你会添加什么？10 倍版本是什么？"

**目标：激发创意，快速构建，享受过程。**

#### 模式切换的智慧

这个设计很重要。

太多工具是"一刀切"。但**不同阶段需要不同的建议**。

创业者需要听到："你的需求证据不足。"

爱好者需要听到："这个想法很酷！试试这个！"

在 gstack 里，我们把这个智慧编码进了系统。


**关键设计：**
- 元数据完整（日期、分支、状态）
- 证据驱动（引用用户的原话）
- 决策记录（方案比较、推荐及理由）
- 上下文继承（`Supersedes:` 字段链接到之前的设计）

### 上下文继承：设计文档的演进

`/office-hours`有个巧妙的设计：**设计文档可以演进**。

每次你在新分支上运行`/office-hours`，它会：
1. 检查是否有现有设计文档
2. 如果有，读取它
3. 在新文档中添加`Supersedes: {prior design}`字段

这样你可以追踪设计如何随时间演进：

```
Design v1 (2026-03-01): 初始想法
    ↓
Design v2 (2026-03-05): 基于用户反馈调整
    ↓
Design v3 (2026-03-10): 增加新功能
```

**这就是工件驱动的上下文传递。**

下游技能（`/plan-ceo-review`、`/plan-eng-review`）会读取这些文档，理解决策历史。

### 创始人信号：AI 如何观察你

`/office-hours`有个元功能：**观察你的思考方式**。

在对话过程中，它会追踪这些信号：
-  articulating a **real problem** someone actually has (not hypothetical)
-  named **specific users** (people, not categories)
-  **pushed back** on premises (conviction, not compliance)
-  their project solves a problem **other people need**
-  has **domain expertise** — knows this space from the inside
-  showed **taste** — cared about getting the details right
-  showed **agency** — actually building, not just planning

然后它会在设计文档中写下：

```markdown
## 我注意到的你的思考方式

- 你没有说"小企业"，你说的是"Sarah，50 人物流公司的运营经理"。这种具体性很罕见。
- 当我挑战前提#2 时，你反驳了。大多数人只是同意。
- 你提到了一个具体的意外：用户把你的产品用于完全不同的目的。那是真正的产品洞察。
```

**这不是奉承，是观察。**

它帮助你看到自己的思考模式，好的和坏的。

### 与下游技能的协同

`/office-hours`不是孤立的。它是整个流程的起点。

设计文档会被下游技能自动读取：

```
/office-hours
    ↓ (设计文档)
/plan-ceo-review  ← 读取设计文档，挑战战略
    ↓ (范围决策)
/plan-eng-review  ← 读取设计文档，设计架构
    ↓ (测试计划)
/qa               ← 读取测试计划，执行测试
```

**每个技能都知道上下文，不需要重新解释。**

这就是为什么 gstack 是"团队"，不是"工具集"。

### 实战建议：如何最大化/office-hours 的价值

基于我的经验，给你几个建议：

#### 建议 1：诚实回答

不要试图"给出正确答案"。

如果你的需求证据不足，就说不足。如果你没有观察过用户，就说没有。

**/office-hours 不是考试，是诊断。**

只有诚实，才能得到有价值的建议。

#### 建议 2：接受挑战

当`/office-hours`挑战你的前提时，不要防御。

思考："它说得对吗？我是否忽略了什么？"

**最好的产品往往诞生于前提被挑战之后。**

#### 建议 3：保存所有设计文档

不要删除"失败"的设计文档。

它们是你的思考历史。6 个月后回看，你会惊讶于自己的成长。

而且，有时候"失败"的设计里藏着真正的机会。

#### 建议 4：邀请团队成员参与

如果你的团队有不同意见，让他们也运行`/office-hours`。

比较生成的设计文档。差异会揭示深层次的假设冲突。

**对齐不是"同意"，是"理解彼此的假设"。**

### 结语：思考的质量决定产品的质量

在 YC，我们常说："如果你解决了错误的问题，做得再好也没用。"

`/office-hours`的目标就是帮你**确保你在解决正确的问题**。

它不能保证你的产品成功。但它能保证：
- 你不会浪费 6 个月做一个没人要的东西
- 你不会忽略明显的风险
- 你不会错过更好的方案

**思考的质量决定产品的质量。**

在 AI 时代，执行的边际成本趋近于零。那思考的价值呢？

**思考的价值被放大了 1000 倍。**

因为错误的思考会导致你以 1000 倍的速度奔向悬崖。

所以，投资思考。

运行`/office-hours`。
