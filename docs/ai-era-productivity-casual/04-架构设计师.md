# 第四章：架构设计师——像工程经理一样思考

## 从"能用就行"到"稳如泰山"

有个团队找我："我们的支付系统有问题，用户被重复收费。"

我看了他们的工作。功能是能用的——用户能付款，钱也能到账。

但架构设计师发现了问题：
- 没有防止重复操作的保护，多个请求同时来会重复扣款
- 没有失败恢复机制，扣款成功但订单创建失败时钱不退
- 没有监控告警，问题发生后3天才发现

**这不是"小问题"，这是"架构缺陷"。**

### 为什么需要架构设计？

在传统工作中，谁负责架构？
- 资深专家/工程经理

他们的价值是什么？
- 能看到别人看不到的边界情况
- 能预见到6个月后的问题  
- 知道什么可以走捷径，什么必须做对

但现在，你一个人就能拥有这种能力！

### 架构设计的六个关键点

1. **数据流** — 数据怎么流动，哪里可能出问题
2. **状态机** — 每个有状态的东西，包括不可能的状态转换
3. **错误处理** — 错误怎么传播，怎么恢复
4. **部署步骤** — 怎么安全地上线
5. **恢复流程** — 出问题时怎么恢复
6. **安全考虑** — 哪里可能被攻击

### 画图真的很重要

别觉得画图浪费时间！一张好的架构图能帮你：
- 理清自己的思路
- 让别人快速理解你的设计
- 发现潜在的问题

而且现在AI能帮你自动生成架构图，超级方便！

记住：**好的架构不是一开始就完美的，而是能随着需求演进的。**
